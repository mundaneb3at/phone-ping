Claude Code → Phone Ping

A free phone buzz — only when Claude is actually waiting on you
attention-only · zero cost

The idea

Kick off a long task in Claude Code, walk away, and get a buzz on your phone only when Claude genuinely needs you — when it asks a question, or when it has sat idle for 60s after finishing. No buzz on every back-and-forth turn. No paid services, no API keys: one PowerShell script, two hook entries in settings.json, one free mobile app (ntfy.sh).

Attention-only redesign (2026-06-13). The earlier version pinged on every turn-end and buzzed twice for a single question. Now: respond within 60s and you get nothing; step away and you get exactly one ping, with a message that says why.

The hooks

PreToolUse
matcher: AskUserQuestion
Claude asks a multiple-choice question. One ping per call.
"Claude has a question"
Notification
matcher: (empty)
Idle 60s after Claude finished (the "you walked away" signal).
"Claude is waiting"
Stop
matcher: (empty)
Fired at the end of every turn.
removed — was the noise
Why drop Stop? Stop fires at the end of every assistant turn, so it buzzed the phone during normal back-and-forth — and a single AskUserQuestion produced two pings (one from PreToolUse before the question, one from Stop at turn-end). With Stop gone, the idle-60s Notification covers the "your turn" case only when you're actually away. Want a buzz on every turn back? Add a Stop entry identical to the Notification one.
Why these two? AskUserQuestion is a tool call, not a notification event — it slips past Notification entirely, so the PreToolUse entry catches it. And in bypassPermissions mode there are no permission prompts, so Notification only ever fires on the idle-60s case — which is exactly what we want.

Setup

  1. Install the ntfy mobile app

    ntfy by binwiederhier on Android or iOS. Free, no account. Open the app and Subscribe to a topic. Topic names on ntfy.sh are a public namespace — use a random suffix so it's not guessable (e.g. my-claude-4829).

  2. Drop the hook script in place

    Copy ping_notify.ps1 to ~/.claude/hooks/ping_notify.ps1. Set your topic near the top:

    $topic = '<YOUR-TOPIC>'   # e.g. my-claude-4829

    The script reads the hook event from stdin to pick the message and debounce duplicates — no other edits needed.

  3. Enable the kill-switch flag

    The script no-ops unless this file exists. This is your on/off control.

    New-Item -ItemType File "$env:USERPROFILE\.claude\ping_enabled" -Force

  4. Wire the two hooks in settings.json

    Merge into ~/.claude/settings.json under the "hooks" key (replace <USER> with your username). Full copy in settings.snippet.json:

    {
  "hooks": {
    "Notification": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }],
    "PreToolUse": [{
      "matcher": "AskUserQuestion",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }]
  }
}

  5. Verify

    Run the diagnostic — it checks the wiring and sends two test pings:

    powershell -NonInteractive -File "$env:USERPROFILE\.claude\ping_check.ps1"

    Expect 2 phone notifications: "Claude has a question", then "Claude is waiting". Then start a new session for the wiring to load.

Troubleshooting

No notifications at all

Confirm the ntfy app is subscribed to the exact topic name (case-sensitive) and that ~/.claude/ping_enabled exists. Run ping_check.ps1 — it pinpoints which layer is missing.

SSL error 35 / cert errors

Don't use curl on Windows in a Bash/Git Bash context — stale system certs cause SSL handshake failures. The script uses PowerShell's Invoke-WebRequest for this reason. Don't swap it back.

Idle reminder fires repeatedly

If Notification re-fires every 60s while you stay idle, raise $debounceSec in the script so reminders don't stack.

I actually want a buzz every turn

Add a Stop entry identical to the Notification one. The script will label it "Claude needs input".

Why not just desktop notifications?

Desktop notifications only work when your laptop is open and unlocked. Phone push works when you've walked away from the desk — which is the actual use case. Kick off a long task, go make coffee, get one buzz when Claude needs you back.
