Claude Code → Phone Ping

One push notification per prompt-cycle — interactive Desktop sessions only
one ping per cycle · zero cost

The idea

Start a long task in Claude Code and receive a push notification when it needs attention — when it asks a question, goes idle, or finishes a turn. At most one ping per prompt-cycle, with a message naming which event fired. No paid services, no API keys: one PowerShell script, four hook entries in settings.json, one free mobile app (ntfy.sh).

Interactive-only. Headless, scheduled, and SDK runs (claude -p, cron jobs, background scheduled tasks) fire the same hooks — but they're gated out, so automation never notifies your phone. Only interactive Desktop sessions ping.

The hooks

UserPromptSubmit
matcher: (empty)
You send a prompt. Resets the per-cycle gate so the next event can ping.
(no ping — gate reset)
PreToolUse
matcher: AskUserQuestion
Claude asks a multiple-choice question.
"Claude has a question"
Notification
matcher: idle_prompt
Claude went idle waiting for user input.
"Claude is waiting"
Stop
matcher: (empty)
Claude finished a turn.
"Claude is done"
One ping per cycle. The first of those events to fire in a prompt-cycle sends; the rest are suppressed until your next prompt resets the gate. The marker is namespaced by session_id (concurrent chats don't suppress each other), self-expires after 30 min, and is written only after a successful POST — so a failed send never silences the next real ping.
Why interactive-only? Stop and the other events also fire for claude -p, cron jobs, and other background runs. Two gates drop those: (a) any run whose cwd is a …\Temp\… dir is skipped (the reliable gate), and (b) only the Desktop app's entrypoint pings — and (b) is fail-open, so a hook subprocess that didn't inherit the env var can never silently drop all pings.

Setup

  1. Install the ntfy mobile app

    ntfy by binwiederhier on Android or iOS. Free, no account. Open the app and Subscribe to a topic. Topic names on ntfy.sh are a public namespace — use a random suffix so it's not guessable (e.g. my-claude-4829).

  2. Drop the hook script in place

    Copy ping_notify.ps1 to ~/.claude/hooks/ping_notify.ps1. Set your topic near the top:

    $topic = '<YOUR-TOPIC>'   # e.g. my-claude-4829

    The script reads the hook event from stdin to pick the message, gate background runs, and debounce per cycle — no other edits needed.

  3. Enable the kill-switch flag

    The script no-ops unless this file exists. This is your on/off control.

    New-Item -ItemType File "$env:USERPROFILE\.claude\ping_enabled" -Force

  4. Wire the four hooks in settings.json

    Merge into ~/.claude/settings.json under the "hooks" key (replace <USER> with your username). Full copy in settings.snippet.json:

    {
  "hooks": {
    "UserPromptSubmit": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }],
    "PreToolUse": [{
      "matcher": "AskUserQuestion",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }],
    "Notification": [{
      "matcher": "idle_prompt",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }],
    "Stop": [{
      "matcher": "",
      "hooks": [{ "type": "command", "command": "powershell -NonInteractive -File \"C:\\Users\\<USER>\\.claude\\hooks\\ping_notify.ps1\"" }]
    }]
  }
}

    If you already have other PreToolUse / UserPromptSubmit / Stop entries, add these as additional objects in those arrays, not replacements.

  5. Verify

    Run the diagnostic — it checks the four-hook wiring and POSTs three test pings:

    powershell -NonInteractive -File "$env:USERPROFILE\.claude\ping_check.ps1"

    Expect 3 phone notifications: "Claude has a question", "Claude is waiting", "Claude is done". Then start a new session for the wiring to load.

Designed, not yet receipt-verified. A green ping_check proves the wiring and that ntfy accepted the POST (HTTP 2xx) — not that the notification surfaced on your phone. Confirm receipt on the device for a real, natural trigger before trusting it (iOS routes ntfy through APNs; check the app subscription, notification permissions, and any Focus / Scheduled-Summary batching).

Toggle on / off

Turn OFF
Remove-Item "$env:USERPROFILE\.claude\ping_enabled"
Turn ON
New-Item -ItemType File "$env:USERPROFILE\.claude\ping_enabled" -Force

toggle_ping.ps1 (in this folder) flips the state in one call.

Troubleshooting

No notifications at all

Confirm the ntfy app is subscribed to the exact topic name (case-sensitive) and that ~/.claude/ping_enabled exists. Run ping_check.ps1 — it pinpoints which layer is missing and writes any send failure to %TEMP%\claude_ping_error.log.

Pings sent (200 / marker) but nothing on the phone

Delivery, not send, is the problem. On iOS, ntfy routes through Apple APNs — check the app is subscribed to the right topic, notifications are allowed, and no Focus mode or Scheduled Summary is batching them.

Transport / TLS

The script POSTs with the Windows System32 curl.exe (Schannel TLS, real exit codes — so failures are logged, not swallowed). Don't swap it back to Invoke-WebRequest (which hid a TLS silent-fail), and don't point it at the MSYS / Git-Bash curl (stale-cert SSL handshake issues).

Automation is buzzing my phone

A headless/cron run slipped past the gates. Confirm it runs from a …\Temp\… cwd (gate a) or sets a non-Desktop CLAUDE_CODE_ENTRYPOINT (gate b). Keep gate (b) fail-open.

Why not just desktop notifications?

Desktop notifications only work when the machine is open and unlocked. Phone push reaches you after you've stepped away from the desk — the actual use case. Start a long task, step away, and receive one notification when Claude needs attention.
